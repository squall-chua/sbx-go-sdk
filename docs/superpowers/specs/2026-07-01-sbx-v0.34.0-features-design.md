# Design: sbx v0.34.0 SDK feature coverage

- **Date:** 2026-07-01
- **Status:** Approved (design), pending implementation plan
- **Targets:** `sbx` / `sandboxd` v0.34.0 (daemon API `0.16.0`)

## Background

The SDK was synced to sbx v0.34.0 (see the `feat!: sync SDK to sbx/sandboxd
v0.34.0` change). v0.34.0 introduced several user-facing capabilities not yet
modelled by the SDK. This design adds coverage for the ones that fit the SDK's
shell-out + REST architecture.

### Candidate features considered

| Feature | Decision | Rationale |
|---|---|---|
| Daemon **settings** (`sbx settings`) | **In scope** | Real gap; generic + reusable; underpins the SSH feature toggle and `kit.allowedSources`. |
| **SSH** access helper | **In scope** | Per-sandbox connection info + enable/provision ergonomics. |
| `sbx setup` wrapper | **Dropped** | Interactive-only TUI, no headless/`--yes` flag — not cleanly automatable. Its value (import agent creds from env into the global store) overlaps the existing `secret.Set` (global scope). |
| Client-side **name validation** | **Dropped** | The daemon's authoritative name rule is not extractable from the binary (built dynamically); a hand-written copy would drift. The daemon validates and its error already propagates. |

## Scope

**In:** two new shell-out packages — `settings` and `ssh` — both using `--json`
CLI output for robust parsing (unlike `policy`/`secret`, which parse rendered
tables because no `--json` exists upstream for those).

**Out:** `sbx setup` wrapper, client-side name validation, kit management,
modelling every `ssh.*` setting individually (they are reachable generically via
`settings`).

## Package 1 — `settings`

Wraps `sbx settings {list,get,set,unset}`. Shell-out (edits `settings.json`,
hot-reloaded by CLI + daemon within ~5s).

```go
package settings

type Setting struct {
    Key         string          `json:"key"`
    Value       json.RawMessage `json:"value"`   // typed: bool/int/float/string/array/object
    Type        string          `json:"type"`    // "bool"|"int"|"float"|"string"|"json"
    Source      string          `json:"source"`  // "default" | "env" | "override"
    Description string          `json:"description"`
}

func List(ctx context.Context, c *client.Client) ([]Setting, error)            // sbx settings list --json
func Get(ctx context.Context, c *client.Client, key string) (*Setting, error)  // sbx settings get --json <key>
func Set(ctx context.Context, c *client.Client, key, value string) error       // sbx settings set <key> <value>
func Unset(ctx context.Context, c *client.Client, key string) error            // sbx settings unset <key>

// Convenience accessors on Setting (values are JSON-typed):
func (s Setting) Bool() (bool, error)   // errors if the value is not a JSON bool
func (s Setting) String() string        // unquoted scalar; raw JSON text for arrays/objects
```

Behaviour:
- `Value` stays `json.RawMessage` so typing is honest; callers can `json.Unmarshal`
  into a concrete type. `Bool()`/`String()` cover the common scalar cases.
- `Get` on an undefined key: the CLI exits non-zero → mapped to sentinel
  `settings.ErrUndefined`.
- `set` parses the value by the setting's declared type (bool/int/float/string/json);
  the SDK passes the value through as a string and lets the CLI parse it.

## Package 2 — `ssh`

SSH-specific ergonomics composed on top of `settings` (feature flag + `ssh.port`)
plus a shell-out to the `sbx ssh setup` provisioning helper.

```go
package ssh

func Enable(ctx context.Context, c *client.Client) error           // settings set feature.ssh true
func Disable(ctx context.Context, c *client.Client) error          // settings set feature.ssh false (explicit, not Unset)
func Enabled(ctx context.Context, c *client.Client) (bool, error)  // settings get feature.ssh -> parse {"enabled":...}

func Setup(ctx context.Context, c *client.Client, opts ...Option) error  // sbx ssh setup [--alias X] [--regenerate]

func Port(ctx context.Context, c *client.Client) (int, error)      // settings get ssh.port (daemon default 2222)

type Target struct {
    User string // sandbox name (SSH username)
    Host string // "127.0.0.1"
    Port int    // ssh.port
}
func TargetFor(ctx context.Context, c *client.Client, sandboxName string) (Target, error)  // reads ssh.port
func (t Target) Args() []string   // ["<name>@127.0.0.1", "-p", "2222"]
func (t Target) String() string   // "ssh <name>@127.0.0.1 -p 2222"

// Options for Setup:
func WithAlias(alias string) Option   // --alias (default "sbx")
func WithRegenerate() Option          // --regenerate (rotate managed key)
```

Behaviour:
- `feature.ssh` is a structured feature-flag; `settings get feature.ssh`
  returns `{"enabled":bool,"variant":...}`. `Enabled` parses `.enabled`.
- `Disable` uses `set feature.ssh false` (explicit) rather than `Unset`, so the
  result is deterministic regardless of the default.
- `Target` uses the raw `127.0.0.1:<ssh.port>` connection form, which works
  without any client provisioning. The `ssh <name>@sbx` alias form is what
  `Setup` provisions into `~/.ssh/config` and is documented as the alternative.
- `Setup` (`sbx ssh setup`) is idempotent and non-interactive; safe to re-run.

## Error handling

- Shell-out failures: `client.MapError` (consistent with `policy`/`secret`/`template`).
- `settings.Get` on a missing key → `settings.ErrUndefined` sentinel.
- `Setting.Bool()` on a non-bool value → a typed parse error.

## Testing

- **Unit** (no daemon): use the existing `recordingClient` fake-`sbx` pattern
  (see `policy/policy_test.go`) —
  - assert exact CLI args for `Set`/`Unset`/`Enable`/`Disable`/`Setup` (incl. option flags);
  - feed canned `--json` stdout to verify `List`/`Get`/`Port`/`Enabled` parsing and
    the `Setting` convenience accessors;
  - cover `Get` undefined-key → `ErrUndefined`, and `TargetFor` Args/String output.
- **Integration** (`//go:build integration`): **read-only** — `settings.List` and
  `settings.Get` against the live daemon (verify the `--json` shape holds). Do
  **not** run `Set`/`Unset`/`Enable`/`Disable`/`Setup` in integration; they mutate
  real `settings.json` / `~/.ssh/config`.

## Docs

- README: add `settings` and `ssh` to the feature list with short usage blocks;
  note the read-only integration-test policy for settings.
- `skills/sbx-go-sdk/SKILL.md`: add both packages to the API surface + a gotcha
  that `settings`/`ssh` mutations are shell-outs that change host state.

## Non-goals / future

- No client-side name validation (daemon is source of truth).
- No `sbx setup` wrapper (interactive TUI).
- Individual typed helpers for every `ssh.*` setting — reachable via `settings`.
